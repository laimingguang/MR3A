# 项目介绍

## 技术栈与流程

本项目基于 **MaaFramework** 开发。脚本核心在 **Pipeline**：用多个节点 JSON 描述状态转移；典型节奏是 **先识别、再点击**，尽量避免 **DirectHit + 点击** 这类「未识别就操作」的写法，以降低误触风险。

## 任务封装与出错回退

MR3A 对任务做了统一封装，并采用 **jump_back** 机制：流程中若节点出错，可 **跳回任务入口** 进行重试

## 配置约定

本项目 **不鼓励** 在节点上长期堆叠 **`rate_limit: 0` + `pre_delay: 0` + `post_delay: 0`** 这类「零节流、零间隔」写法；**仅非动作节点**（无 `action` 的纯识别/跳转节点）才适合把 **`pre_delay` / `post_delay`** 配成 `0`，其余节点请保留合理节流与间隔，避免打满模拟器与识别。

## 已知限制

作者能力与时间有限，**3v3 跑图** 当前以 **固定延时点击** 为主，对 **不同延迟、不同网络环境** 的设备 **兼容性一般**；若你环境差异较大，需自行微调或接受偶发偏差。

## 远端静态配置（版本文案、兑换码等）

项目通过 Agent 中的 [`remote_json_override_expected.py`](../../agent/custom/action/remote_json_override_expected.py)（注册名 **`RemoteJsonOverrideExpected`**），经 **jsDelivr** 等镜像拉取 GitHub 仓库里的 **静态 JSON**（如 `assets/remote/game-meta.json`），在运行时 **合并进流水线**：例如更新 OCR 的 **`expected`**，或 **`InputText`** 的 **`input_text`**，从而在不发整包更新的情况下同步版本相关文案与兑换码等字段。
